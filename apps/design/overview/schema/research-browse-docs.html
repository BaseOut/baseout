Schema Browse & Docs — research & build decisions

Grounding the two new Schema tabs in evidence: what Browse (a searchable explorer of every base/table/field) and Docs (long-form documentation that tags real entities) actually are, the problems they solve, how the best schema-catalog and document tools build them, and the patterns we should adopt, avoid, and decide on before building. This is the reference to lean on — now and when questions come up later.

Companion to the client briefs overview/schema/04-browse-tab.md and 05-docs-tab.md (and the page README). Compiled 2026-06-24 from three parallel research passes — schema-catalog teardown, drill-in-panel & doc-editor teardown, and a Mobbin visual sweep — plus first-party verification of the Airtable metadata API and the Plate / React Flow APIs.

TL;DR

  • Browse = a data catalog over one Space's Airtable schema. The two proven shapes for it are dbt docs' Project↔Database sidebar toggle (= our Tree ↔ Flat-index toggle) and Stripe's API reference field row (name (type · nullable · linked) → description). Build both verbatim.
  • The detail panel is a real push/pop stack modelled on iOS UINavigationController: slide-in-from-right on drill, breadcrumb in the header, back pops one, and the Miller-columns rule — drilling a different branch truncates the stack past the branch point. Keep it ≤ ~35–40% of the viewport (Jira's "35%+12px" is the cautionary upper bound). The list stays live on the left (Notion side-peek), zero context loss.
  • Docs = Notion/Coda-style entity @-mentions over the schema. Typing @ inserts a live, identity-referencing entity-tag chip (not a text snapshot) that opens the same shared detail panel; tagging is bidirectional — each tagged entity lists the docs that reference it. Embedded mini ER-diagrams are saved React-Flow views (the exact Fibery-Mermaid-in-a-doc pattern).
  • Production editor = Plate (first-class mention combobox, custom inline-void node, edit-vs-read rendering, slash menu). For this dependency-free mirror we ship a high-fidelity editor shell that nails the same UX and is annotated with the Plate target. See §8.
  • Three things share one build: the entity detail panel (Browse + Docs), the entity search/typeahead (Browse search + Docs @ + Visualize add-entity), and the entity tag chip (the identity used everywhere). Build each once.
  • Anti-fabrication is the hard constraint. Airtable's meta API gives base/table/field names, types, descriptions and link targets — and nothing about record counts, value statistics, workspace grouping, or usage/popularity. Every column and panel section is scoped to that boundary (§2).

1What we're building, and the problem it solves

The Schema section is where Baseout stops being "just a backup tool." Five tabs, one job — understand the Space's Airtable structure — asked five ways: Browse (find & inspect), Visualize (the diagram), Changelog (what changed), Health (what's wrong), Docs (write it down). This report covers the two new ones. New tab order: Browse · Visualize · Changelog · Health · Docs.

Browse

find & inspect

The job: "Let me find and inspect any base, table, or field fast, and see everything attached to it." A power user who knows roughly what they want shouldn't have to pan a diagram. They want a searchable, navigable list and a detail panel.

The shape: a browse area (Tree or Flat index) + one global search + faceted filters, and a right-side stacking detail panel that drills base → table → field → linked table without losing your place.

Docs

write it down

The job: "Let me write up how this part of the schema works — referencing the actual tables and fields, with a diagram and some links — and have those references stay live and clickable."

The shape: a documents list + a document view (rich editor, inline @-entity tags, a tags panel, external links, saved mini-diagrams) + a clean reading mode. A document tags entities; those tags surface back on each entity in Browse.

Descriptions vs documents — the distinction the UI must hold. A description (Browse) is one short field on an entity (Airtable's, AI-generated, or yours). A document (Docs) is a page you write that can span many entities, with diagrams and links. Don't conflate them: a description is a field; a document is a page. The connective tissue between all three tabs is the entity tag — the same identity used by Browse rows, Docs chips, and Visualize nodes.

2The Airtable data boundary load-bearing

Verified directly against Airtable's developer docs. This governs every column and panel section we can honestly draw. Browse and Docs read the same schema the engine captures during a backup, so what the metadata API does and does not return is the boundary on what's real.

SourceWhat it gives us (real)What it does NOT give (do not fabricate)
GET /meta/bases Per base: id, name, permissionLevel. No workspace / workspaceId grouping. No table counts.
GET /meta/bases/{id}/tables Per table: id, name, description?, primaryFieldId, fields[], views[]. Per field: id, name, type, description?, options (e.g. link's linkedTableId / inverseLinkFieldId / prefersSingleRecordLink). No record/row counts. No per-field value statistics (distinct %, null %, min/max). No usage / popularity / "who queried it."
Backup snapshot (our engine) Record counts, change history, health score — computed at backup time, labelled "as of last backup." Drifts between runs. Never present as live; always stamp the freshness.

Do-not-fabricate flags

  • Record counts only exist from backup data → label "as of last backup," never a live count.
  • Column profiling (distinct/null/min/max) is impossible from metadata → no stats panel. (Atlan/Castor have it because they sit on the warehouse; we don't.)
  • Workspace grouping isn't returned → the "Workspace" column stays a placeholder; don't group by it.
  • Popularity / usage / "referenced by" from query logs → we have none. Our only honest "references" signals are schema-internal: linked-record relationships (multipleRecordLinks + computed-field options) and doc tags (which we author). Both are real; use those.

One upside of the boundary: the full Base → Table → Field tree arrives in one call per base, so Tree and Flat are both built client-side with no extra fetching, and the "linked" relationship is fully resolvable (linkedTableId → click straight to the target table, with real data).

3Browse: how the best schema explorers do it

Eleven schema-catalog / data-dictionary tools, teardown. The two most directly stealable are dbt docs (the Tree/Flat toggle and peek-on-select) and Stripe's API reference (the field-row format).

dbt docs / dbt Catalog

closest analog
Tree / flat
The toggle to copy: a left-sidebar Project ↔ Database switch — Project = logical folder tree, Database = warehouse hierarchy. dbt Catalog expands to Resources / File-Tree / Database tabs and renders a selected type as a sortable table in the main pane. This is exactly Tree↔Flat, proven.
Search
Searches name, description, column names, SQL, tags with relevance scoring; results filterable by resource type; fuzzy/partial, -exclude, boolean OR, quoted phrase; filters panel appears after searching.
Detail
Peek on select, full page on commit: selecting a node opens a side panel (summary); double-click / search-click opens the full page. Tabs General / Code / Columns; status bar (last run, rows, size); explicit "Depends on / Referenced by" sections; per-column inline ✅ test status.
Steal: the Project/Database (= Tree/Flat) toggle top-left; peek-on-select; the Referenced-by / Depends-on labelled sections; the status bar (rows/size/last-run) as trust signals. Avoid: the classic one-giant-scrolling-page — dbt itself moved to tabs.

Stripe API reference

the canonical field row
Layout
Three-pane: left resource nav, middle flat attribute list, right sticky JSON example. Nesting by indentation + dot-path, not a tree.
Field row
mono name → muted (type, nullable, expandable) → plain-English description. Modifiers stack. Enums get a "Possible values" block. Nested objects expand inline behind "Show child attributes."
Steal: the name (type · nullable · linked) + description row — maps 1:1 onto an Airtable field. Pair the detail with a live example. Avoid: no search inside a long attribute list (fine for read-once docs, risky for a repeat-use tool) — add filter-as-you-type over the field list.

DataGrip · DBeaver · TablePlus

the search gold standard
Search
DataGrip two-tier: (a) speed-search type-to-highlight with abbreviation matching (fa→film_actor); (b) a command-palette "Go to DB object" (Ctrl+N) across the whole catalog, expandable to include columns/indexes. Active-filter indicators (green dot, DBeaver's (…)).
Icons
DataGrip stacks overlay modifiers on a base glyph (gold key = PK, blue key = FK, NOT-NULL dot). DBeaver shows type label inline.
Detail
DBeaver's tab set — Properties / Columns / Keys / Foreign Keys / Indexes / DDL — is the cleanest canonical table-detail IA.
Steal: two-tier search (filter-the-tree + Cmd/K go-to-object, abbreviation-aware); active-filter indicators so users never think data is missing. Avoid: DBeaver's "filter only applies to expanded nodes" — the classic "I searched and it wasn't there" trap. Make search hit the whole schema.

Notion database side-peek

the panel reference
Detail
List stays live on the left, page opens on the right — zero context loss. Modes Side / Center / Full. Layout: title → stacked property name/value list (each with a muted type icon) → body. Controls clustered top-left (expand-to-full, close).
Linked drill
Relation value → opens the linked page (itself a peek). True peek-within-peek stacking is undocumented; observed behaviour is replace-content + breadcrumb back.
Steal: list-left / detail-right zero-context-loss; properties as a stacked name/value list with type icons; controls top-left. Adapt: we're read-only post-backup → drop the edit affordances; one default panel mode, not three.

Also surveyed (full notes in the source agents): Atlan (Cmd/K, live count-grouped-by-type, three-dot column show/hide, type-specific stat blocks — but a steep many-tabs learning curve to avoid); Secoda ("/" palette with Recent/Popular default, column-header → sort/add-filter, nested search, Source-path scoping); Select Star (three-zone table page, popularity-ranked default, A→Z / Z→A / Ordinal sort, join-indicator icon); Castor (persistent right rail that survives tab switches, a documentation-completion % with itemised breakdown, source-logo provenance); Metaphor / DataHub (field-scoped search operators name: / fieldPaths: / quoted / -negation); Hasura (object-vs-array relationship naming = Airtable single-vs-multiple links); AWS Glue (colour reserved for change-state — a finished design for our Changelog); Retool (schema-on-demand toggle, one box matching tables + columns, a visible Refresh + staleness signal).

4Browse: the design we'll build

The 7 load-bearing decisions

  1. Tree ↔ Flat toggle, top-left (dbt). Tree = Base ▸ Table ▸ Field; Flat = a real sortable data table of every entity (Name / Type / Parent / Linked / Health), default sorted health ascending (worst-first — trust-signals-first, our stance, adapting Select Star's popularity-first).
  2. Two-tier search (DataGrip): a filter-as-you-type that narrows the visible Tree/Flat in place, plus a Cmd/K palette that jumps to any base/table/field (incl. field names) — abbreviation-aware. This palette is the same entity-typeahead reused by Docs @ and Visualize add-entity.
  3. Stripe field row: icon + name + muted (type · linked?) + description, split into real columns in Flat. Right-align counts; offer A→Z / Z→A / Ordinal (source order) sort.
  4. One Lucide icon per Airtable field type (our Lucide-only rule — NOT Airtable's proprietary glyphs; the vendored field-icon set is the documented exception, used where recognisability wins). Mapping table below.
  5. Faceted filter bar reusing our shipped FacetFilter pattern, with facets we can actually back: base · entity type · field type · health · missing description · has AI description · linked. Show "Showing 142 of 1,310" + a Clear, so a filtered list never reads as missing data (DataGrip/DBeaver active-filter indicator).
  6. Notion side-peek detail panel: list live on left, panel right, zero context loss, read-only. Sections per §5.
  7. Schema staleness up front (Retool): "Schema as of {last backup}" near the toggle; any count labelled the same way.

5The stacking detail panel shared component

Used by both Browse (row click) and Docs (tag click, in reading mode). Build it once. The model is iOS UINavigationController, not modal-on-modal.

Panel contract

  • Anchor + size: right-anchored overlay, width ≈ 30–40% of viewport, min ~420px, with collapse/close. The browse list stays interactive on the left (Notion). Jira's resizable "35%+12px" default — and the user complaints when it balloons to ⅔ — is the cautionary upper bound.
  • A real stack of sheets: clicking a child / related entity pushes a sheet that slides in from the right (~150–200ms ease-out); back / Esc / ← pops one. Never a second floating modal over the panel.
  • Breadcrumb trail in the header reflects the stack (Base › Customers › Status); click a crumb to pop to that depth; collapse the middle with "…" so the header never overflows on deep stacks.
  • Miller-columns truncation rule: if you're at base→table→fieldA and drill a different related table, sheets past the branch point are discarded — one coherent path, no orphaned crumbs.
  • Reading-mode reuse: the same component opens read-only when a Docs entity-chip is clicked.

Panel sections (per the brief, scoped to real data)

  1. Context + type — name, type badge, breadcrumb of where it lives, key stats (field/record counts "as of last backup", health dot). For fields: the field-type icon + the type.
  2. Descriptions — Airtable's (read-only, what's live now) · AI description (Generate / Regenerate, Pro+, 10 credits) · Yours (editable) + Copy AI → Yours · Push to Airtable shown disabled / "Coming soon" (V2 write-back). This block matches the Visualize side-panel exactly.
  3. Children — base → tables; table → fields; field → options (select choices). Each clickable → pushes its sheet.
  4. Relationships — field → its linked/lookup/rollup target; table → tables linked via fields. Clickable. Single-vs-multiple naming from prefersSingleRecordLink (Hasura's object-vs-array idea).
  5. Documentation — the docs (from the Docs tab) that tag this entity, clickable to open. The reverse side of Docs tagging; "No documents reference this yet" otherwise. honest references

6Docs: how the best entity-mention editors do it

The contract every good mention system satisfies — Notion, Coda, GitHub, Linear, Craft — has four parts. Our entity tag must meet all four.

The entity-tag contract

  1. Identity, not snapshot — store the entity id, render its live current name. Rename the table → every chip updates (Notion).
  2. Clickable — opens the target (Coda: "click a chip to learn more about that object") → our shared detail panel in reading mode.
  3. Hover preview — a lightweight hovercard with type + status (GitHub hovercards; Craft's preview card right in the @ dropdown) so you confirm without committing.
  4. Bidirectional — referencing an entity creates a reverse "referenced by" backlink on it (Notion, GitHub). This is the high-value bit; it powers Browse's Documentation section.

Notion / Coda

@-mentions + backlinks

@ + title → live dropdown → inline chip with the page's icon + title; rename keeps every mention in sync; automatic backlinks ("Linked to this page", click jumps to the exact spot). Coda's chip → "learn more about that object" is precisely our click-to-detail.

GitHub

references + hovercards

# autocompletes; references auto-create a reverse backlink; hovering shows a hovercard with status/title — the cheap preview to steal.

Embedded mini-diagrams

dbt · Fibery · Whimsical · Eraser
dbt Explorer
Lineage embedded in each doc page; a mini-map (bottom-right) expands to the model's immediate parents/children, colour-coded by type. The strongest analog: a scoped diagram (just this entity's neighbours) living inline, expandable.
Fibery
A Mermaid block renders an actual node-and-arrow diagram inside the page — the single most on-point reference for our saved mini-ER-diagram.
Whimsical / Eraser
Embed = small saved preview inline → expand/fullscreen to inspect → view-only in the doc, edit in the source. Eraser/dbt go further: live-updating, multiple per page.
Steal: a saved diagram block that renders read-only in the doc with an Expand-to-fullscreen affordance that opens the full Visualize canvas (Craft's whiteboard-block → fullscreen). Multiple named diagrams per doc; each an independently saved React-Flow view.

7Docs: the design we'll build

  • List view (PandaDoc / ClickUp): Title · last edited · tag count · excerpt · a row of linked-entity chips; search/filter (reuse FacetFilter); a New document button. Empty state: "No documents yet — document how a base or workflow is structured." (quiet, no hero illustration).
  • Document view: editor (main) + a tags panel (rail) + Links & Diagrams. A header "Link entity" affordance (ClickUp) in addition to inline @.
  • Inline @-tagging: typing @ opens the shared entity-typeahead, grouped by kind (Bases / Tables / Fields) with a Craft-style preview (type · parent · health) so the author picks the right one. Selecting inserts a tag chip. In reading mode the chip → the shared detail panel.
  • Tags panel: every tagged entity listed, add/remove here too (not only inline) — the explicit association list that populates Browse's Documentation section. Soft semantic badges per our interface rules.
  • Links: named external references (name + URL), visually distinct from entity chips so "internal entity" ≠ "external URL."
  • Diagrams: one or more saved React-Flow mini-views (scoped tables, saved toObject() blob = nodes/edges/viewport). Read = static, pannable, Expand-to-fullscreen; edit = the canvas. Multiple per doc.
  • Reading mode (Craft / Notion): clean centred column, author + last-updated byline, an auto Table-of-Contents rail mirroring headings, a right minimap for long docs; tags clickable → panel; diagrams read-only/expandable.
  • Removed-entity state: a tag whose entity is gone from Airtable stays, flagged "no longer in schema" — never silently dropped.

8The editor decision dependency fork

The brief names Plate because the editor must render custom inline nodes (the entity tags), which a plain textarea can't. Plate covers every requirement first-class — verified from its docs:

  • Mention combobox: MentionPlugin + MentionInputPlugin with trigger: '@', triggerPreviousCharPattern, and a selection handler that inserts the node and moves the cursor.
  • Custom entity-tag node: an inline void element { isElement, isInline, isVoid } carrying { entityId, entityKind, value }, declared via withTriggerCombobox + @platejs/combobox. The chip is a contentEditable=false span; useReadOnly() / useSelected() switch its rendering between edit and reading mode.
  • Slash menu (SlashPlugin) to insert blocks — including an ER-diagram void block (Plate's editable-void pattern hosts a live React Flow canvas inside contentEditable=false).
  • Markdown round-trip: mentions serialise to [Customers](mention:tbl_123) (entity id as the URL), so docs persist cleanly.

Decision — mirror vs production

Plate is a heavy dependency, and this ui-only repo must boot locally with zero backend and is sensitive to the registry minimumReleaseAge gate. Adding it here is a real engineering choice that belongs to the monorepo, not the design mirror.

Plan: ship a high-fidelity, dependency-free editor shell for the mirror — a contenteditable surface wired to the same entity-typeahead, real inserted tag chips, the tags panel, links, and an embedded React-Flow mini-diagram (reusing the existing SchemaCanvas) — so the entire UX (the @ flow, chips, reading mode, diagrams) is reviewable now. The editor core is annotated with the Plate target and the concrete plugin map above, so the monorepo swap is a contained, well-specified change. This is honest to "no fabricated dependency" and keeps the mirror runnable.

10Adopt / avoid, consolidated

Adopt

  • Tree↔Flat toggle (dbt); Flat = real sortable table, health-ascending default.
  • Two-tier search: filter-in-place + Cmd/K go-to-entity (abbreviation-aware).
  • Stripe field row; Ordinal sort option; right-aligned counts.
  • Reuse our shipped FacetFilter; "Showing N of M" + Clear.
  • Notion side-peek (list-left/detail-right) + iOS push/pop stack + breadcrumb + Miller truncation.
  • Entity-tag four-part contract (identity / clickable / hover / bidirectional).
  • Fibery/dbt scoped inline mini-diagram, expandable to the full canvas.
  • Castor documentation-completion % (we have the inputs); staleness stamp (Retool).

Avoid

  • Stats / profiling panels — impossible from Airtable metadata.
  • Live record counts or workspace grouping — fabrication.
  • Modal-on-modal; a panel wider than ~½ the viewport (Jira's complaint).
  • Search that only hits expanded nodes (DBeaver trap) or silently hides no-permission items (Glue) — "empty" must differ from "hidden".
  • Atlan's many-tab overload for a focused base→table→field tool.
  • A consumer/publishing tone in Docs — keep it utility-admin dense (no decorative blocks, no exclamation marks).
  • Adding Plate to this mirror repo (belongs in the monorepo).

11Build plan

Order follows the brief's own "design first" — the shared pieces, then Browse, then Docs composes on them.

  1. Shared entity model + typeahead — flatten the fixture schema into a single entity index (base/table/field) with kind, parent, type, health, description-state. One EntitySearch typeahead (Browse filter + Cmd/K + Docs @ + Visualize add-entity).
  2. Shared stacking detail panel (EntityPanel) — the push/pop stack, breadcrumb, 5 sections, read-only + reading mode. New storybook entry + catalog component.
  3. Browse tab — Tree + Flat (reuse Table + toolbar pattern), FacetFilter bar, staleness stamp, wired to the panel.
  4. Docs tab — list view; document view (editor shell + @ + tags panel + links + mini-diagram via SchemaCanvas); reading mode. Plate annotated as the production target.
  5. Harness + handoff — extend the schema.astro fixture with descriptions, doc fixtures (tags + a saved diagram + links), wire the 5-tab bar (Browse default), update flow-registry.ts + storybook (entity panel, typeahead, tree row, tag chip).

12Open decisions for the client

  1. Field-row iconography: reuse the vendored Airtable field-icon set (recommended — already on Visualize/Changelog, max recognisability) vs. standardise on the Lucide mapping (§4) for strict single-icon-set purity. recommend: vendored set
  2. Health vocabulary on rows: a simple green/amber/red dot vs. Braintrust-style named states (Verified / Incomplete / Deprecated / Needs review). recommend: dot now, named states later
  3. Docs gating tier — brief says "likely Pro+; confirm with the capability matrix."
  4. Diagram edit depth in Docs — saved view-only (Whimsical model, simpler) vs. fully editable inline (Confluence model). recommend: view-only + "open in Visualize" to edit
  5. Plate now or later — confirm the mirror-shell approach (§8) is acceptable for review, with Plate landing in the monorepo.
