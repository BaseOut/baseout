Schema Visualize — research & build decisions

Grounding the Schema diagram in evidence: how competing schema / ER tools actually render and interact with a diagram, whether React Flow is the right engine, and the patterns we should adopt, avoid, and decide on before building. This is the reference to lean on — both now and when questions come up later.

Companion to the client brief in overview/schema/ (README + per-tab docs). Compiled 2026-06-22 from two parallel research passes (competitor teardown + library pressure-test) plus visual references on Mobbin.

TL;DR

  • Engine: React Flow (@xyflow/react v12) is confirmed. It is the only library that satisfies all four hard constraints at once — custom HTML/JSX node cards, first-party React, hierarchical auto-layout, and read-only click-to-inspect. Validated by Liam ERD's formal ADR, ChartDB, and Supabase Studio's own schema visualizer.
  • Layout: @dagrejs/dagre (the React-Flow-team-recommended default; small, synchronous). Already installed. elkjs only if we later need sub-flows or finer edge routing.
  • One real design fork to settle (§4): fields-in-node vs compact-node + side-panel. The client brief picked compact + panel; every ERD tool we studied — including Airtable's own extension — puts field rows in the node with field-to-field edges. Recommendation: a hybrid.
  • Steal from Airtable's own "Base Schema" extension (the On2Air/Openside-lineage incumbent we're displacing): its colour-by-relation-type edge language. Beat its weaknesses: hover-only highlight, no search, no auto-layout.
  • Hard "don'ts": no force-directed network graph (even Airtable avoids it), no render-everything-at-once, no crow's-foot cardinality (misrepresents Airtable's two-way / many-to-many links).

1Engine — React Flow, confirmed

The decisive constraint is the custom HTML/JSX node card (a table with a health dot, name, counts, and — depending on §4 — field rows). That single requirement splits the field cleanly:

  • Canvas / WebGL renderers (Cytoscape.js, vis-network, Sigma.js, AntV G6 in fast mode) draw nodes as shapes + text + images. You cannot render a styled React field-list card inside a node. They are also built for force-directed social graphs — the wrong shape of problem.
  • DOM / SVG renderers (React Flow, Reaflow, GoJS, raw D3) can host a real component tree. React Flow goes furthest: a node literally is your React component.

React Flow also ships the rest of the checklist as first-class built-ins: <MiniMap/>, <Controls/>, <Background/>, pan/zoom, and per-field connection <Handle>s. xyflow even publishes an official "Database Schema Node" component (a prebuilt table node with a labeled handle per column) — i.e. they endorse this exact use case as a first-class pattern. MIT-licensed, very actively maintained.

What React Flow deliberately leaves to us (plan for these)

None are blockers; they are the work we own. For a read-only viewer it's a favourable trade — we skip all editing complexity and mostly need layout + custom rendering.

  1. No built-in auto-layout. RF's docs say plainly: "We have not implemented our own layouting solution yet." → wire in dagre (§2). The single biggest "missing battery."
  2. No crow's-foot / cardinality edges. RF ships 4 edge types (bezier, smoothstep, step, straight). Relationship markers need a custom edge with SVG markers. For us this is a feature, not a gap — we deliberately don't want crow's-foot (§5).
  3. Field-level handles are manual. Anchoring an edge to a specific field row means rendering a <Handle> per field with a unique id. The official Database Schema Node demonstrates it; we wire it per row.
  4. It's a toolkit, not a finished ER tool. We assemble nodes / edges / layout / markers. drawDB-level polish is our code.
  5. Astro: runs as a client-side React island (client:only="react") — needs the DOM, won't SSR meaningfully. Expected. already wired & verified

Validation — tools built on React Flow

Verified from package.json / ADRs, with honest counter-examples:

  • Liam ERD — a formal ADR (2024-11-12) choosing React Flow over a custom canvas; cites 100+-table perf, built-in pan/zoom, ELK layout. Strongest single datapoint.
  • ChartDB — @xyflow/react ^12.8.2 in package.json. Supabase Studio's official schema visualizer — built on React Flow. Also Dify, DBDraw, TSDiagram, Hubql (RF showcase, DB/schema cluster).
  • Honest counter-examples: drawDB uses a bespoke SVG/foreignObject renderer (proof a custom renderer is viable, just more work); n8n uses Vue Flow (a separate library — don't cite as RF evidence).

2Auto-layout — dagre, not elkjs (to start)

React Flow has no layout of its own, so we add one. The two real options:

dagre (@dagrejs/dagre)elkjs
Quality (hierarchical/ER DAG)Very good for trees/DAGsBest-in-class; port constraints, edge routing, sub-flows
ConfigurabilityMinimal — a virtue hereHuge (Eclipse Layout Kernel, Java-ported) — needs the Java API ref open
SpeedFast, synchronousSlower; runs async (web worker)
Bundle~tens of KB~500 KB (ported Java)
RF team stance"We highly recommend dagre" for trees"We don't often recommend elkjs… difficult to support"
MaintenanceActive as @dagrejs/dagre (v2/v3, late 2025)Active (v0.11)

Verdict: dagre. RF-team-recommended for hierarchical layouts, an order of magnitude smaller, synchronous, trivial to configure. Reach for elkjs only if dagre's edge routing/overlap proves inadequate at ~200 nodes, or we need sub-flow/grouped layouts (dagre has a known sub-flow gap) or fine port-based anchoring.

Gotcha — install the scoped package

The "dagre is deprecated" claims online refer to the old unscoped dagre package. The maintained fork is @dagrejs/dagre (DagreJs org), shipping through late 2025. we installed the right one

3Competitor teardown — what each uses, where, and how

Each card: how the node is built, how edges work, what drives layout, the key interactions, the tech, and the one thing worth stealing. Confidence noted where a claim is inferred.

Airtable "Base Schema" extension

the incumbent we displace@airtable/blocks · leader-line · svg-pan-zoom · React 16

Same data model, same On2Air/Openside lineage. Its dependency list is the decisive evidence on the force-directed question.

Node
Tables with their fields laid out; hovering a field shows its type in a bottom-left tooltip. Drag to rearrange.
Edges
Colour-coded by relation type, hover-activated: Linked record = solid gray (bolds on hover); Formula = animated dashed blue; Rollup = dashed purple; Lookup = dashed orange; Count = dashed red. Each appears on hover of the relevant field.
Layout
No graph-layout library at all — lines drawn field-DOM-element → field-DOM-element via leader-line; positioning is manual/drag. Pan/zoom via svg-pan-zoom. Not force-directed (verified from source).
Interactions
Hover a field → its relations light up. That's mostly it.
Tech
Airtable Blocks SDK + leader-line + svg-pan-zoom + lodash. No d3-force, webcola, dagre, elk, or React Flow.
Steal: the relation-type colour language. Beat: hover-only (nothing persists), no search/jump, no auto-layout — those three gaps are our whole opening.

Azimutt

best philosophy for "faster to scan"open source
Node
Does not show all fields — on big schemas an entity shows only its ~15 most important attributes, ordered PK → relations → indexes → SQL order. Entities collapse to header-only while keeping relationship lines.
Edges
Relationship endpoints as small coloured link icons on the table's left (incoming) / right (outgoing) edges; you navigate by clicking them. Relations of hidden entities/attributes are suppressed to cut clutter.
Layout
Rejects "render every table at once." Many named, curated layouts; each starts empty and you build it via "search everywhere" + "follow relations" / "find path".
Interactions
Search across tables/columns/relations/constraints/comments; follow-relations; shortest-path between two tables; folder-organised layouts. Engineered for 1500 tables / 100k columns.
Tech
Custom (not React Flow); its value is the model, not the renderer.
Steal: "don't show everything — let the user pull in what's relevant": search-everywhere, follow-relations, show-only-key-fields, collapse. The single best idea for beating the source app.

React Flow "Database Schema Node" (the reference building block)

adopt the anatomy@xyflow/react + shadcn/ui
Node
Header = table name. Body = one row per field, two columns: field name (left) + field type (right).
Edges
Per-field connection handles — every field row carries a target handle (left) + source handle (right). Edges connect via sourceHandle/targetHandle matching field ids → genuinely field-to-field (e.g. warehouse_id → id).
Layout
Whatever you wire (dagre/elk).
Tech
Official xyflow component — the cleanest, most directly adoptable pattern for our stack.
Steal: the exact node anatomy + per-field handles. This is xyflow officially endorsing our use case.

ChartDB

@xyflow/react ^12.8
Node
Field rows (name + type) with PK / nullable / unique indicators; fields rendered conditionally for perf on big tables.
Edges
Field-to-field; cardinality markers at each end via SVG marker defs.
Layout
Auto-arrange is a manual "Reorder Tables" button ("minimise overlapping") + it highlights overlapping tables. No dagre/elk named.
Tech
React Flow + React context + browser storage.
Steal: overlap-minimise + flag overlaps; conditional field rendering for perf.

dbdiagram.io

DBML
Node
Field rows (name + type, PK/FK markers); driven by DBML text.
Edges
Field-to-field refs; crow's-foot cardinality (its house notation).
Layout
Table Groups — collapse/expand, custom colours, notes; per-relationship colours. "Tailor the level of detail."
Interactions
Double-click a table/column → jump to its definition (two-way navigate). Zoom; no minimap found.
Steal: Table Groups (collapsible coloured clusters) for chunking big bases; two-way jump.

drawDB

custom SVG (no flow lib)
Node
Tables with columns, indexes, FKs; drag-and-drop.
Edges
Clean orthogonal (elbow) connectors; drag from a blue dot on the FK column to the PK column → field-anchored.
Layout
cautionary tale historically no auto-layout — SQL import dumped all tables stacked on top of each other; hand-sorting required.
Steal: orthogonal elbow connectors. Lesson: skipping layout is a real failure mode — for a read-only viewer, auto-layout is mandatory.

dbdocs.io

DBML, read-only docs
Node
Read-only published table pages; expand a node to reveal its relationships.
Edges
"Relationship" tab = full ERD; per-table "Table references" list.
Layout
Positions saved & persist across visits.
Steal: closest in intent (read-only, documentation). Persist the layout so a returning user sees the same arrangement.

Metabase & Retool — data-model browsers (not diagrams)

the "list sometimes beats a diagram" lesson
Metabase
No spatial diagram. Tree browser (DB → schema → tables → columns); each table page lists a "Connections" section of FK relationships as clickable links; X-ray auto-profiling.
Retool
Left-panel searchable schema browser (tables + columns/types); partial fetch for large schemas.
Lesson
For many real users a fast searchable list with relationship links beats a diagram — worth offering the side panel / search as a first-class companion, not an afterthought.
Steal: make the searchable list + relationship links a first-class companion to the canvas (this is also our Changelog/side-panel surface).

Least relevant (freeform canvases, not structural ER engines): Lucidchart (crow's-foot library, auto-draws on DB import, manual), Miro (ERD templates, whiteboard), Excalidraw (sketch aesthetic, weak schema model). Not models for us.

4The one real design decision — node anatomy

This is the fork worth deciding deliberately, because the evidence diverges from the current brief.

School A — fields-in-node (true ERD)

Node = header + field rows (name · type icon · PK/link markers). Edges anchor field-to-field.

Who: React Flow Database-Schema-Node, ChartDB, dbdiagram, drawDB, Airtable's own extension.
Pro: relationships are legible at the node; it's a real ER diagram.
Con: denser nodes; needs progressive disclosure on big bases.

School B — compact node + side-panel

Node = health dot + name + counts. Click → side panel with the full field list.

Who: Neon & Supabase table browsers (list + panel, not a canvas).
Pro: dense, uniform, scan-first; matches the brief's "not a dashboard of mini-charts."
Con: the diagram shows boxes-and-lines only; you can't see which fields relate without clicking.

Recommendation — a hybrid

The brief picked B; the ERD evidence is unanimous for A (even Airtable's extension). Airtable bases are smaller than enterprise DBs, so density is manageable. Proposed middle path:

  • Node shows the relationship-bearing rows — the primary field + linked-record fields (the rows that actually carry edges), with type icons. Edges anchor to those specific field rows (School A's payoff) — so the diagram is a real ER view, not just boxes.
  • Everything else collapses — a node can show header-only by default on large bases (Azimutt), expanding to its key rows; the full field list + Airtable types + the AI "Generate description" live in the side panel (School B's payoff + the brief's panel intent intact).
  • Health dot + counts stay on the node header exactly as the brief specifies.

5Patterns to adopt & anti-patterns to avoid

Adopt (read-only, structural Airtable schema viz)

  1. Field-to-field edges anchored on the specific linked-record row, not table-to-table — that's the information Airtable users actually need.
  2. Colour/encode edges by Airtable relation type (linked / lookup / rollup / formula / count) using Airtable's own colour language — not crow's-foot.
  3. Structural auto-layout (dagre, layered) + fit-to-view, with orthogonal (elbow) connectors (drawDB). Allow drag to nudge; persist the layout (dbdocs).
  4. Click a table → persist-highlight its relationships and dim the rest (focus mode). Fixes Airtable's hover-only weakness; matches Azimutt's follow-relations.
  5. Search-everywhere + jump-to-table (Azimutt / Retool) — the highest-leverage single feature for "faster to scan than Airtable itself."
  6. Progressive disclosure — collapse to header-only; key-fields-first; table groups (dbdiagram) to chunk big bases.
  7. Standard canvas affordances — fit-to-view, zoom/pan, minimap, dotted-grid background.
  8. A first-class searchable companion list (Metabase/Retool) — our side panel + (later) Changelog already lean this way.

Avoid (confirmed against real source, not opinion)

  • Force-directed "network graph" (floating bubbles, springy edges, edges-everywhere). No serious ERD tool uses it — not even Airtable's own extension (it uses leader-line + manual positioning). The brief warns against this; the source confirms it.
  • "Render every table at once" — Azimutt's whole thesis is this fails past ~30–50 tables. Less acute for Airtable, but the principle holds: progressive over exhaustive.
  • No auto-layout (drawDB's import-stacking bug). Read-only means we own the arrangement — auto-layout is mandatory.
  • Hover-only relationship reveal with no persistent state (Airtable's gap) — you can't study a relationship while reading both tables.
  • Crow's-foot cardinality. Correct for FK databases, semantically wrong for Airtable — linked records are often two-way and many-to-many by default; forcing 1:N would misrepresent the model. Use Airtable-native semantics.

6Mapping to our design system

The canvas + custom node + health cards are bespoke Patterns (documented in the storybook), but the chrome leans on catalog primitives, and any new shared piece goes into the design system (per our build rule).

  • Visual register: closest reference on Mobbin is Vapi's dark, dense node canvas — dotted grid, bottom floating zoom/fit toolbar, minimap bottom-right, right-side inspector. Match that density, on our tokens.
  • Side panel (the field list): the model is Neon's table-detail panel — table header + field rows (icon · name · type), grouped, monospace for types (our mono-boundary rule). Field detail like Supabase's column panel. Build it on the catalog Drawer/side-panel + Table primitives.
  • Node card: a React component, styled with our CSS tokens (--color-base-100/300, --radius-box) so it themes light/dark with the app. Health dot = --color-success/warning/error (identical to the Health tab band, so the two tabs reinforce).
  • Field-type icons: the vendored Airtable set in overview/schema/field-icons/ (31 SVGs, currentColor) — used in the node rows and the side panel. The documented exception to Lucide-only.
  • Chrome: tabs-box (Visualize/Changelog/Health), header base-filter (Select), Export menu (Button), empty/gated states — all catalog primitives.

7Build plan — what RF gives vs what we own

React Flow gives us (free)

  • The canvas: pan / zoom / fit-to-view
  • <MiniMap/>, <Controls/>, <Background/>
  • Custom node = our React component
  • Per-field <Handle>s for edge anchoring
  • Selection / hover state, light/dark colorMode

We build (the three custom pieces)

  1. Layout wiring — dagre → positions → RF nodes; fit-to-view on load; persist nudges.
  2. Custom edge — orthogonal connector + colour by Airtable relation type (no crow's-foot).
  3. Per-field handles — anchor edges to specific linked-record rows.

Increments (Visualize first, per the brief): (1) page shell — header + tabs-box + states; (2) canvas with custom table nodes + dagre + minimap + a rich schema fixture; (3) field-to-field edges + relation-type colour; (4) side panel (field list + AI generate-description states); (5) focus-mode + search; (6) Health & Changelog tabs. The React Flow island is already wired and verified (/schema-test).
