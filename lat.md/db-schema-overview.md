# DB Schema Overview

The **master DB** holds organisational metadata, billing, connections, Spaces, and subscriptions. Customer Airtable record data lives elsewhere, in per-Space client DBs.

Per-Space client DBs (D1 / Shared PG / Dedicated PG / BYODB) are provisioned by `apps/server`. Source for this section: [master-schema.ts](../master-schema.ts) and [shared/Master_DB_Schema.md](../shared/Master_DB_Schema.md). Frontend (`apps/web`) is the canonical owner of master-DB migrations; backend mirrors what it needs.

## Core Tables

The minimum set of tables you'll touch when reasoning about Organizations, Spaces, Connections, and billing. Each row in this list is an `export const` in [master-schema.ts](../master-schema.ts).

| Table | Purpose |
|---|---|
| `platforms` | Seeded reference (Airtable, Notion, HubSpot, …) with short prefix codes (`at`, `nt`, `hs`) |
| `organizations` | Top-level billing entity. One Stripe customer per row. |
| `organization_members` | User ↔ Organization membership and role |
| `spaces` | Container bound to one Platform. Primary unit of config + billing. |
| `space_platforms` | Resolves which platforms a Space exposes (V2-aware) |
| `at_bases` | Airtable Bases tracked within a Space (platform-prefixed table per [[db-schema-overview#DB Schema Overview#Platform-Prefixed Tables]]) |
| `connections` | OAuth/API-key link between an Organization and a Platform. Holds encrypted tokens. |
| `connection_sessions` | Per-Connection rate-limit session state mirrored from the ConnectionDO |
| `subscriptions` | Stripe subscription per Organization |
| `subscription_items` | One row per active platform/tier within the subscription |
| `overage_records` | Audit of credit/storage overage events tied to a billing period |
| `user_preferences` | Per-user UI/notification settings |

Backup, restore, storage, webhook, notification, and health-score tables ship in subsequent migrations.

## Naming Conventions

The convention from [shared/Baseout_PRD.md](../shared/Baseout_PRD.md) §21.2. Apply these everywhere; do not mix camelCase or PascalCase into DB-facing names. Drizzle column names in TypeScript are camelCase by convention but always map to snake_case in SQL.

| Convention | Rule | Example |
|---|---|---|
| Table names | Plural, snake_case | `organizations`, `spaces`, `backup_runs` |
| Column names | Singular, lowercase, snake_case | `organization_id`, `created_at`, `is_active` |
| Primary keys | `id`, UUID | `id uuid primary key default gen_random_uuid()` |
| Foreign keys | `{singular_referenced_table}_id` | `organization_id`, `space_id` |
| Timestamps | `created_at`, `modified_at` | Set on insert; `modified_at` on every change |
| Boolean fields | `is_` / `has_` prefix | `is_active`, `has_migrated` |
| Enum-like status | snake_case strings | `'pending' \| 'running' \| 'success' \| 'failed'` |
| Encrypted columns | `_enc` suffix | `oauth_access_token_enc` (AES-256-GCM, see [[security-model]]) |

## Platform-Prefixed Tables

Tables holding platform-specific data are prefixed with the platform's short code from `platforms.code`. This keeps multi-platform schemas clean — no ambiguity between an Airtable base and (eventually) a Notion page.

- Airtable (`at`): `at_bases`, future `at_tables`, `at_views`, etc.
- Notion (`nt`, V2+): `nt_pages`, …
- HubSpot (`hs`, V2+): `hs_objects`, …

## Master DB vs Client DBs

Two-tier model — **master DB is single-tenant control plane; client DBs are per-Space data planes.** Keep them mentally separate; never join across them.

- **Master DB (this file)** — DigitalOcean shared PG (Trial/Launch/Growth/Pro Organizations) or Neon/Supabase dedicated (Business+). Holds `organizations`, `spaces`, `connections`, billing.
- **Client DBs** — provisioned per Space, tier-dependent (D1 / Shared PG / Dedicated PG / BYODB). Hold customer record data. Schema generated by `apps/server` from the Airtable Base schema.

## Mirrored Tables in `apps/server`

`apps/server` is **not** the migration owner. It mirrors specific tables (e.g. `backup_runs`, `backup_configuration_bases`) so the Worker can read/write them without crossing the network for every operation.

Each mirrored Drizzle file in `apps/server/src/db/schema/` carries a header comment naming the canonical migration source in `apps/web`. See [apps/server lat graph](../apps/server/lat.md/) `db-mirror.md`.

## Migrations

Drizzle Kit generates and applies migrations. Migration files are committed alongside the schema change. Production never auto-applies — review precedes every run.

- Generate: `drizzle-kit generate` (in `apps/web`)
- Apply: `drizzle-kit migrate` (per environment)
- Mirrored files in `apps/server/src/db/schema/` are kept in sync **manually** when the canonical migration changes.
