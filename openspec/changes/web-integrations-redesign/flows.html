Integrations — user flows & states

Every state a user can be in on the Integrations screen, and every path between them — from an empty Space through connecting Airtable, configuring protection, and recovering a broken connection. The source of truth for these flows is specs/integrations/spec.md; each diagram region maps to a requirement and its scenarios in the table below.

B Connect flow — OAuth vs API key

From the empty hero, the user picks a connection method. Both end in base discovery on success; every failure returns to the hero with a specific, human-readable reason.

flowchart TD
    Start(["Connect hero"]) --> M{"Method"}
    M -->|"OAuth (recommended)"| OA["Redirect to Airtable consent"]
    M -->|"API key"| AK["Paste personal access token
(read-only scopes)"]
    OA --> OAr{"Result"}
    OAr -->|approved| Disc["Discover bases"]
    OAr -->|access_denied| E1["Error — you declined access"]
    OAr -->|missing_code| E2["Error — no auth code returned"]
    OAr -->|token_exchange_failed| E3["Error — Airtable rejected the connection"]
    OAr -->|api_call_failed| E4["Error — connected, but base list failed"]
    AK --> AKr{"Token valid?"}
    AKr -->|yes| Disc
    AKr -->|no| E5["Error — invalid or wrong-scope token"]
    Disc --> OK["Connected — found N bases"]
    OK --> Cfg[["Configure flow → C"]]
    E1 --> Start
    E2 --> Start
    E3 --> Start
    E4 --> Start
    E5 --> Start

C Configure & protect

The configuration panel, top to bottom: choose bases, choose layers, set a schedule, pick destinations (static and dynamic can run in parallel), then run the first backup. Branch nodes are the edge cases — tier cap, zero bases, plan-locked frequency, BYOS auth, broken connection.

    Bx --> B

D Connection health — recovery state machine

How a live connection moves between health states over its lifetime, and how the UI responds. Refreshing is transient and keeps backups running; pending_reauth and invalid are the recovery states that route back into the connect flow.

stateDiagram-v2
    [*] --> active: connection established
    active --> refreshing: token auto-refresh
    refreshing --> active: refreshed (backups keep working)
    active --> pending_reauth: token expiring / revoked
    refreshing --> pending_reauth: refresh failed
    pending_reauth --> active: user reconnects
    pending_reauth --> invalid: left unresolved
    invalid --> active: user reconnects
    active --> active: new bases discovered (banner)

    note right of pending_reauth
        Amber. Backups paused.
        Reconnect = primary action.
    end note
    note right of invalid
        Red. Backups will not run.
    end note
