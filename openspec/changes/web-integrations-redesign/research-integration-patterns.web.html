★ The canonical heavy-config flow (closest domain analogs)

Fivetran and Airbyte move data out of a source on a schedule to a destination — exactly Baseout’s shape. Both put config on a dedicated guided flow, never a modal, and both end in an explicit verify step. Our flows.html diagram C matches this.

Takeaways for us

  1. The ordered flow Connect → pick data → schedule → destination → verify is the industry standard.
  2. “Save & Test” / verify-after is a first-class step — frame our “run first backup & confirm” as that verify beat.
  3. Save-for-later / resumable partial setup (Fivetran) is worth copying for first-time users.

⇉ Parallel destinations — our most distinctive requirement

The industry does fan out to multiple destination types at once. Treat destinations as independent, separately-authenticated objects, and let one backup write to several — exactly our static + dynamic in parallel.

⏱ Scheduling & the “connection” data model

Scheduling: a frequency picker with discrete options plus cron for power users (Fivetran: 1 min–24h). Plan-gate higher frequencies as visible-but-upgradeable, not hidden — consistent with how these tools sell tiers.

Data model: Airbyte’s shape — reusable sources + reusable destinations, a connection binds them — implies authenticating a source once and reusing it is the expected mental model. Direct evidence for resolving our open question toward Org-level Connect (auth once, reuse across Spaces).

Sources: Fivetran connections · Airbyte connections · Airbyte destinations

A Naming & the marketplace→settings pattern

Naming: across these tools the recurring word is “sync” / “connection”; “backup” is reserved for point-in-time snapshots. Baseout does both — so the naming split is real, not interchangeable. Evidence for resolving our open question deliberately.

Vercel / Stripe: install from a marketplace → land on the integration’s settings page (API keys as env vars, rotate keys, Connect Project). Reinforces: connected → a dedicated settings page; and it surfaces an API-key path, like our delta.

✓ Implication (both sources converge)

  1. Overview = card (status + quick actions) → flips empty → connected.
  2. Heavy config = dedicated, tabbed route with an explicit Save & Test / verify step; resumable.
  3. Edit = same detail page, sections in place — no wizard re-run.
  4. Destinations = independent objects, fan-out in parallel, each with its own auth sub-step.
  5. Connect likely Org-level (auth once, reuse) — take to the client as evidence on the scope question.
