### New drift items introduced by the pulled changes

#### A. R2 was removed but downstream proposals still assume R2 exists

Commit `8fc1f61` removed every R2 binding from `apps/server`. CSVs now write via `local-fs-write.ts`. But:

- `baseout-backup-attachments/proposal.md` Phase A still names `r2_object_key` as a column on `attachment_dedup` and Phase B says "pipe to R2".
- `baseout-backup-byos-destinations/proposal.md` Phase B lists `r2-managed.ts` as a strategy that "wraps the existing `env.BACKUPS_R2.put` path." That path is gone.
- `baseout-backup-retention-and-cleanup/proposal.md` says "the cleanup engine issues `DELETE` requests" against R2 objects under `r2://baseout-backups/<spaceId>/<runId>/`.

**Possible explanations:**

1. R2 removal was a temporary measure to unblock local-dev iteration; R2 returns when destinations land. The proposals assume the eventual state.
2. The product pivots to a purely BYOS model where there is no managed R2 default — Baseout never holds customer bytes.
3. The proposals predate the R2-removal commit and need updating.

**Recommendation:** before any of these proposals start implementation, reconcile the R2 question in writing. If (1), the `byos-destinations` change needs a Phase 0 step to "re-introduce the managed R2 binding under the StorageWriter interface." If (2), the `attachments` + `retention-and-cleanup` proposals need rewrite to drop R2 specifics. If (3), the proposals just need an edit pass.

#### B. `apps/hooks` and `apps/api` become real Workers

- `baseout-backup-instant-webhook` puts the webhook receiver in `apps/hooks`.
- `baseout-backup-automations-interfaces-docs` puts the inbound API in `apps/api`.

Both apps are currently skeleton stubs. Each will need:

- Wrangler config + DNS (`hooks.baseout.com`, `api.baseout.com`)
- Service binding from `apps/web` (for the API token validation surface) and into `apps/server` (for the engine forwarding)
- Their own `INTERNAL_TOKEN` parity rules + secret provisioning

This is a non-trivial setup task that should land before the instant-webhook or automations changes start implementation. Worth filing a focused `baseout-hooks-bootstrap` + `baseout-api-bootstrap` ahead of them.

#### C. Engine-side credit-cost reporter ambiguity

`baseout-backup-manual-quota-and-credits` Phase D says the engine reports `attachmentBytes`, `basesProcessed`, `triggerSource` in `/runs/complete`, and apps/web computes credit cost. `baseout-backup-attachments` reports `attachmentsDownloaded` count (different unit).

Reconcile: the engine should emit both (`attachmentCount` and `attachmentBytes`) so apps/web has what it needs for both Trial cap enforcement (count) and credit cost (bytes).

#### D. Dependency chain is now explicit but unenforced

The new proposals reference one another by name in their Out-of-Scope tables, forming this rough DAG:

```
schedule-and-cancel (PARTIAL)
  ├─→ attachments (depends on trial-quota for Phase D)
  ├─→ instant-webhook (depends on dynamic-mode)
  ├─→ retention-and-cleanup (depends on manual-quota for Phase D)
  ├─→ byos-destinations (extends r2-managed pattern → see drift A)
  ├─→ trial-quota-enforcement
  ├─→ manual-quota-and-credits
  ├─→ dynamic-mode
  └─→ automations-interfaces-docs (depends on dynamic-mode for write path)
```

There's no enforcement layer for this. Worth either:

- Adding `blocks: [<other-change>]` metadata to each proposal's frontmatter, or
- Adopting a "no implementation start until upstream is at Phase Done" review rule.
